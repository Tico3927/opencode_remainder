## 1. 系统资源简介

### 1.1 计算资源（节点配置）

| 项目 | 配置 |
|------|------|
| **登录节点** | 主机名 `ln07` (64 CPU核 + 256GB 内存)，已联网，仅用于登录/编辑/编译，不要直接运行作业 |
| **GPU 计算节点** | 命名规则 `g0599~gxxxxx`，不能联网 |
| **GPU 型号** | NVIDIA RTX5090 × 8 卡/台，32GB 显存 |
| **资源配比** | 1卡 : 8CPU核(物理核) : 126GB 内存 |
| **单作业卡数** | 1~8 卡 |
| **超核** | ❌ 不支持超核配置 |

### 1.2 存储资源（文件系统）

| 目录 | 用途 | 配额 |
|------|------|:--:|
| `~` (家目录/home) | 环境设置文件 | 1GB |
| `~/run` | 计算数据存放，读写性能好，在此目录计算 | 300GB |

> 超过配额联系客户经理增加。

## 2. 应用及软件管理

### 2.1 已部署软件（Module 管理）

系统使用 `module` 工具统一管理软件环境。

#### 命令速查

| 命令 | 功能 |
|------|------|
| `module avail` | 查看可用软件列表 |
| `module load cuda/12.8` | 加载软件 |
| `module show cuda/12.8` | 查看软件环境详情 |
| `module list` | 查看已加载软件 |
| `module unload cuda/12.8` | 移除加载 |
| `module --help` | 更多用法 |

> module 加载只在当前窗口有效，退出失效。可写入 `~/.bashrc` 永久生效。

#### 2.1.3 已部署软件列表

| 软件 | 版本 |
|------|------|
| miniforge | 25.3.0-3 |
| gcc | 9.3.0, 12.5.0, 13.4.0 |
| CUDA | 12.8, 12.9 |
| 其它 | 持续更新 |

### 2.2 软件安装 / 环境搭建

基本流程：
1. 上传软件安装包
2. 打开 **SSH 或 putty** 命令行窗口，cd 到对应目录，安装/编译

> ⚠️ 文档原文："打开 【SSH】 或者 putty 命令行窗口" 说明集群支持 SSH 远程连接。

#### 2.2.1 Miniforge 自定义 Python 环境

```bash
# 1. 加载 miniforge
module load miniforge/25.3.0-3

# 2. 创建环境 (python 3.10)
conda create -n py310 python=3.10

# 3. 查看环境
conda env list

# 4. 激活环境
source activate py310

# 5. 取消环境
conda deactivate

# 6. 查看已安装包
conda list
```

#### PyTorch 安装（重要）

5090 依赖 cuda12.8，需要 PyTorch 2.7.0+：

```bash
pip3 install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu128
```

⚠️ **PyTorch 2.7.0+ 需要 Python 3.9+**

⚠️ pip 安装的 nccl 用 cuda12.2 编译，与 5090 不兼容（nccl2.25 及以下均不兼容 5090），必须手动用 cuda12.8 编译 nccl2.26.2 覆盖：
```
conda 环境路径/lib/python版本/site-packages/nvidia/nccl/lib/libnccl.so.2
```

## 2.3 提交计算任务

### 计费规则
- 费用 = 卡数 × 时间(精确到秒) × 单价
- 超核惩罚：费用 = 卡数×时间×单价 + (申请CPU核数-8×卡数)×时间×1/8×单价

### 2.3.1 单节点任务提交

```bash
sbatch -p gpu --gpus=GPU卡数 脚本名
```

卡数范围 **1~8**，每卡默认分配 8 核 + 126GB 内存。

#### SLURM 脚本模板 (Python)

```bash
#!/bin/bash
#SBATCH --gpus=1

# 加载环境
module load miniforge/25.3.0-3
source activate py310

# 运行程序
python train.py
```

#### SLURM 脚本模板 (HPC/srun)

```bash
#!/bin/bash
#SBATCH --gpus=2

module load cuda/12.8

# 每卡 1 进程 8 线程
srun -n 2 -c 8 ./your_program
```

#### gpu 队列参数解析

| 参数 | 说明 |
|------|------|
| `-n <number>` | 进程数，默认 1 |
| `-c <ncpus>` | 每进程 CPU 核数，默认 1 |
| `--gpus=卡数` | 申请的 GPU 总卡数 |
| `--gres=gpu:卡数` | 每节点 GPU 卡数（多机推荐） |
| `-p gpu` | 指定 gpu 队列（默认也是 gpu） |

> -n、-c 不设置时默认均为 1；-n×-c < 8×卡数 也会全部分配核心；超核报错。

### 2.3.2 跨节点任务提交

```bash
sbatch -N 2 -p gpu --gres=gpu:8 --qos=gpugpu 脚本名
```

| 参数 | 说明 |
|------|------|
| `-N <node>` | 节点数 |
| `--gres=gpu:卡数` | 每节点 GPU 卡数 |
| `--qos=gpugpu` | ⚠️ 跨节点必须指定，需管理员开通权限
| `--ntasks-per-node=<n>` | 每节点进程数 |

#### NCCL 跨节点通信配置

```bash
export NCCL_DEBUG=INFO
export NCCL_IB_DISABLE=0
export NCCL_IB_HCA=mlx5_bond_0
export NCCL_SOCKET_IFNAME=eth0
export NCCL_IB_GID_INDEX=3
```

### 2.3.3 包节点 VIP 队列

专属队列名格式：`vip_gpu_用户名`

```bash
sbatch --gpus=8 -p vip_gpu_用户名 ./run.sh
```

### 2.3.4 内存文件系统（提速）

将数据集放到内存 `/dev/shm` 减少 IO 瓶颈，最大 504GB：

```bash
# 1. 打包数据集
tar -cf datasets.tar datasets

# 2. 脚本中解压到内存
tar -xf datasets.tar -C /dev/shm

# 3. 代码中改路径
# 改为 /dev/shm/datasets
```

```bash

```

| 列 | 含义 |
|------|------|
| JOBID | 唯一作业号 |
| ST | **R**=运行(计费), **PD**=排队, **CG**=退出中, **CD**=完成, **F**=失败 |

## 2.6 程序测试流程（salloc 交互式）

用于测试/调试，正式任务不要用此方法。

```bash
# 加载 openssl
module load openssl/3.0.2

# 申请 GPU（开始计费！）
salloc -p gpu --gpus=1
# → 输出：Nodes g0599 are ready

# 直接 SSH 到计算节点
ssh g0599

# 查看 GPU 使用率
nvidia-smi

# 查看 CPU 使用率
top

# 释放资源（停止计费）
scancel 作业ID
```

⚠️ salloc 弊端：
1. 资源使用时长与任务运行时长无关，容易忘记取消导致浪费
2. 程序跑完作业不会自动停止，必须手动 scancel
3. 终端断开 = 任务中断
4. 多作业同机只能看到最后一个
